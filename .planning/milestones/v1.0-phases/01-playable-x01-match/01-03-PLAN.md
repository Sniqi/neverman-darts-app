## Phase Goal

**As a** darts player at home, **I want to** play a full X01 match by touch from setup to finish, **so that** I can score a real living-room game quickly and accurately.

> This is the core play-loop vertical slice: it replaces the Plan 01 placeholder match route with the real scoring view — tap the board, see the score drop, get checkout hints, correct mistakes, win the leg. It consumes the tested engine from Plan 02 and the scaffold from Plan 01. After this plan a real user can actually score darts by touch.

<objective>
Build the complete scoring view at `/match`, wiring every input component to the real `matchStore` from Plan 02. Task 1 delivers the core play loop (board tap → score updates → compact checkout suggestion beside the score → undo) so the screen is genuinely usable, with the responsive portrait/landscape layout (D-02). Task 2 adds the refinements that the acceptance criteria require: numpad alternative input with validation, the darts-at-double dialog, the post-visit correction window (D-05), the match-win overlay, and the wake lock.

Purpose: This is where touch scoring becomes real. Output: A fully playable scoring view (portrait + landscape), all input components, wake lock, and a browser-mode hit-detection test.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pointer event (untrusted coords) → board math → reducer | Tap coordinates are transformed to a scored segment; precision errors could misattribute scores |
| numpad digits (user int) → reducer | Player-typed visit total enters the state machine via the UI |
| browser wake-lock API → app | External permission-gated API that may reject |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Numpad accepting an impossible/invalid total | mitigate | `Numpad.svelte` calls `isValidVisitTotal` before dispatch; invalid input shakes and is rejected with no state change (RESEARCH V5 Input Validation). |
| T-03-02 | Tampering | SVG hit-detection drift under scaling misattributing scores | mitigate | Use `screenToBoard` (getScreenCTM().inverse()) + polar `classifyHit`, never SVG `contains()` (Pitfall 4). Browser test asserts center/triple/miss mapping. |
| T-03-03 | Denial of Service | Wake-lock request rejected (power saver) crashing the match view | mitigate | `acquireWakeLock` is wrapped in try/catch and no-ops on unsupported/rejected (RESEARCH V7 Error Handling); match remains fully playable. |
| T-03-04 | Spoofing/XSS | Player name rendered in ScorePanel / win overlay | mitigate | Names rendered via Svelte `{...}` interpolation only (auto-escaped); no `{@html}`. |
| T-03-SC | Tampering | npm installs (supply chain) | accept | No new packages introduced — components built on Plan 01/02 deps already vetted in RESEARCH Package Legitimacy Audit (all Approved). |
</threat_model>
