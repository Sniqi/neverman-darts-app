<objective>
Close the correction-window and numpad-wiring defects that make the scoring view unusable past
the first visit. Four coupled defects in the input layer:

1. CR-03 — `CorrectionWindow` has no `ondismiss` prop. It dispatches `CONFIRM_VISIT` internally
   but cannot tell the parent to clear `pendingCorrection`, so the overlay permanently covers
   the board/numpad after the first visit. The parent's `dismissCorrection()` is dead code.
2. CR-04 — the parent's visit-detection `$effect` compares each player's visit count against a
   single cross-player `lastVisitCount`, so the window only fires for the first visit of the
   entire match and never again. It must track per-player visit counts.
3. ENG-04 (UI half) — because of CR-03/CR-04 the bust correction overlay never clears, so after
   a bust the next player's UI is permanently blocked even though the engine reverted correctly.
4. CR-05 — `Numpad` dispatches `NUMPAD_VISIT` directly to `matchStore`, bypassing the parent's
   `handleNumpadVisit`, so the `DartsAtDoubleDialog` (INP-03) can never appear on a leg-winning
   numpad visit.

Additionally, the current E2E test masks CR-03/CR-04 with `if (await overlay.isVisible()...)`
conditional handling — it must be hardened so the correction-window regression cannot hide again.

Purpose: Make the correction window appear-and-dismiss for every visit (INP-04, D-05), free the
UI after bust (ENG-04), and route numpad visits through the parent so darts-at-double works (INP-03).
Output: A `CorrectionWindow` with an `ondismiss` prop, a `match/+page.svelte` with per-player
visit tracking and proper dismiss wiring, a `Numpad` with an `onconfirm` prop, and an E2E test
that asserts the correction window's lifecycle.
</objective>
