<design_decision>
The naive widening — gating the prompt on a bare `!isFullscreen` — was rejected: it would render the large amber prompt on EVERY non-fullscreen `/display`, including a PC second window during an active match, obstructing the live scoreboard. The two entry points are genuinely different intents, and they reach `/display` by different mechanisms already present in `SpectatorChooser.svelte`:

- PC: `openSecondWindow()` → `window.open(`${base}/display`, '_blank')` (a fresh popup window, opener nulled).
- Tablet: `goToDisplayFullscreen()` → `goto(`${base}/display`)` (same-tab in-app navigation).

Chosen approach: carry the tablet's explicit intent through the navigation as a URL query flag. `goToDisplayFullscreen()` navigates to `${base}/display?fullscreen=1`; the PC path is left unchanged (no flag). `/display` reads the flag once at mount (the route is `ssr:false`, browser-only) and renders the prompt when `!isFullscreen && (idle/setup OR tablet-fullscreen intent)`.

Why this is the simplest correct option:
- Zero new dependencies, zero new cross-window state, no BroadcastChannel involvement.
- Deterministic: the flag encodes exactly the user's choice ("Anzeige hier im Vollbild"), not a fragile heuristic about window type.
- Self-contained: the PC window never carries the flag, so the regression the design consideration warns about cannot occur.
- Existing controls are untouched: `.fullscreen-toggle` (always rendered), `.exit-btn`, and `handleDisplayTap` keep their current behavior.

Read the flag synchronously at mount via `window.location.search` (the intent is fixed for the page lifetime; no reactive `page` store is needed). Do not add `$app/state` / `$app/stores` imports.
</design_decision>
