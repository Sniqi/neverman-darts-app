## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL param `[id]` → DB query | `/history/[id]` id arrives as a string from the URL and is used to fetch a record |
| Stored MatchState → DOM | Player names and stats from persisted matches are rendered in list rows and the detail view |
| match-complete write → IndexedDB | Completed match snapshot is written to the matches table |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Tampering | `/history/[id]/+page.ts` id parsing | mitigate | `parseInt(params.id, 10)` + `isNaN` guard → 404 before any query; Dexie `.get()` returns undefined for unknown keys → 404 (RESEARCH Pitfall 5) |
| T-03-05 | Tampering | HistoryRow / detail rendering stored player names | mitigate | All names/stats rendered via Svelte `{interpolation}`; no `{@html}` in any new component (auto-escaped) |
| T-03-06 | Denial of Service | `db.matches.add` failing in private/quota-restricted mode | accept | `#persistCompletedMatch` wrapped in try/catch — the match was still played; history persistence is best-effort by design (consistent with ensureDbOpen graceful-degradation precedent). No data corruption: a failed add is atomic |
| T-03-SC | Tampering | npm installs | mitigate | This plan adds NO new packages; legitimacy gate not triggered |
</threat_model>
