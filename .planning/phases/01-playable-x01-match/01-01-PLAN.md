## Phase Goal

**As a** darts player at home, **I want to** play a full X01 match by touch from setup to finish, **so that** I can score a real living-room game quickly and accurately.

> Source: ROADMAP.md Phase 1 Goal. This Walking Skeleton plan proves the full stack end-to-end (scaffold → routing → real Dexie read/write → one wired dartboard tap) before the engine and full UI are built in later plans.

<objective>
Scaffold the greenfield SvelteKit 2 + Svelte 5 project per the CLAUDE.md locked stack and prove the entire stack works with the thinnest possible vertical slice: a setup route that writes and reads a player profile from Dexie/IndexedDB, and a match route with one tappable region wired through a runes store that updates a displayed number. Also lay down the test harness (Vitest unit + browser projects, Playwright) and a single failing end-to-end happy-path spec.

Purpose: Establish the architectural backbone (framework, DB, routing, directory layout, test infra) that every subsequent vertical slice builds on. This is the Walking Skeleton — the smallest meaningful capability that exercises the full stack.
Output: A runnable SvelteKit app, a Dexie profiles table, a placeholder dartboard tap interaction, a configured multi-project test runner, a failing E2E spec, and SKELETON.md recording the architectural decisions.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user input (profile name) → DOM render | Player-typed name is rendered back into the setup list; untrusted text crosses into the DOM |
| user input (profile name) → IndexedDB | Player-typed name is persisted via Dexie |
| build config → GitHub Pages static host | `BASE_PATH` controls asset/route base under the repo subpath (Phase 6 deploys; config established here) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing/XSS | Profile name rendered in setup list | mitigate | Render names only via Svelte `{...}` template interpolation (auto-escaped). No `{@html}`, no `innerHTML`. (RESEARCH Security Domain: XSS via player name) |
| T-01-02 | Denial of Service | Dexie `db.open()` in private browsing / quota exhaustion | mitigate | Wrap DB access in try/catch; on open failure the app continues without persisted profiles (guests still work). Surface an inline non-blocking error, never throw to a crash. |
| T-01-03 | Tampering | Build base path misconfiguration leaking wrong asset URLs | accept | `paths.base` is read from `BASE_PATH` env; verified at Phase 6 deploy. Low risk in Phase 1 (local-only); no secrets involved. |
| T-01-SC | Tampering | npm/playwright installs (supply chain) | accept | All Phase 1 packages were vetted in RESEARCH "Package Legitimacy Audit" — all Approved (official sveltejs/vitest-dev/microsoft orgs), zero `[ASSUMED]`/`[SUS]`/`[SLOP]` requiring a checkpoint. No new packages beyond the audited list are introduced. |
</threat_model>
