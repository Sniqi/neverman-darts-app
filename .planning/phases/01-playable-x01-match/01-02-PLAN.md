## Phase Goal

**As a** darts player at home, **I want to** play a full X01 match by touch from setup to finish, **so that** I can score a real living-room game quickly and accurately.

> This plan builds the load-bearing logic core — the pure reducer engine and board math — that every UI slice (Plans 03, 04) wires into. It is a dedicated TDD plan because every unit is pure input→output (state machine, bust math, checkout lookup, polar geometry) where `expect(fn(input)).toBe(output)` can be written before the implementation. The MVP walking skeleton (Plan 01) already proved the stack; this plan makes the rules real.

<objective>
Implement the complete X01 darts engine as a framework-free, exhaustively unit-tested pure reducer over an append-only event log, plus the SVG-board polar-math hit classifier and the numpad validation set. The event-log design is the single mental model for unlimited per-dart undo (D-06). The checkout module returns one route per score, recalculated live after every dart (D-10), using a standard published pro checkout table (D-11). Wrap the reducer in the real Svelte 5 class store (`matchStore`) that Plans 03 and 04 consume — with its own unit test covering dispatch, activePlayer, and the suggestion getter (checker fix: the store contract is behavior-verified, not just type-checked, before downstream plans build on it). No DOM, no Svelte components except the thin store wrapper — everything here is testable in Node.

Purpose: The reducer is the single source of truth for bust handling, undo, leg/set counting, turn rotation, dart tracking, and checkout suggestions. Getting it correct and fully tested BEFORE building UI is the primary recommendation in RESEARCH.md and de-risks the hardest logic early.
Output: `src/engine/*` modules with matching `.test.ts` files (all green), and `src/stores/match.svelte.ts` + `src/stores/match.svelte.test.ts` exposing and verifying `matchStore`.
</objective>

export type MatchAction =
  | { type: 'START_MATCH'; config: MatchConfig; players: Omit<PlayerState,'remaining'|'legsWon'|'setsWon'|'visits'>[]; order: string[] }

  | { type: 'NUMPAD_VISIT'; total: number; dartsUsed?: 1|2|3; dartsAtDouble?: number }
  | { type: 'CONFIRM_VISIT' }

| Boundary | Description |
|----------|-------------|
| numpad total (user int) → reducer | Untrusted integer enters the state machine; must be validated before it mutates score |
| dart input → reducer | Board taps produce DartScore objects; malformed values must not corrupt state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | Invalid numpad total (NaN, negative, >180, impossible) reaching the reducer | mitigate | `NUMPAD_VISIT` reduction calls `isValidVisitTotal()` and returns state unchanged on failure; reject non-integers (RESEARCH Security Domain V5 Input Validation). |
| T-02-02 | Tampering | Mutable state corruption causing stale/incorrect scores | mitigate | Reducer returns a fresh object every call; derived counts recomputed from the event log, never incremented in place (RESEARCH anti-pattern). Unit test asserts immutability. |
| T-02-03 | Information Disclosure | Non-serializable state breaking Phase 2 BroadcastChannel | accept | Engine has no IO and no functions/classes inside MatchState; serializability is enforced by the type shape. Phase 2 verifies channel posting. |
| T-02-SC | Tampering | npm installs (supply chain) | accept | No new packages introduced in this plan — pure TS over the Plan 01 scaffold. All deps already vetted in RESEARCH Package Legitimacy Audit (all Approved). |
