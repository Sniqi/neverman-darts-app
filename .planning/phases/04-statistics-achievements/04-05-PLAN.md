<objective>
Deliver real-time personal-record detection and the celebration overlay across both views (ACHV-01, ACHV-02, ACHV-03). Hook record detection into `MatchStore.dispatch` (the existing broadcast point), preload each profile player's records from `db.matches` at match start (D-09 recompute, via 04-03's `computeLifetimeStats`), broadcast record events over a NEW separate `BroadcastChannel('neverman-record')` to avoid the DisplayStore type collision (RESEARCH Pitfall 5), and render a transient auto-dismiss `RecordOverlay` on `/match` and `/display`. When a record coincides with a leg/match win, fold it into the existing win banner instead of stacking overlays (D-08). Per D-02 the spectator view stays minimal — the record-celebration overlay (D-08) is the ONLY new element added to /display; no stat tables or drawers are introduced there.

After this plan a player can do something new: score a 180 or set a personal record and see it celebrated live on both the input and spectator screens, with the record permanently reflected in lifetime stats (because records are derived from the now-persisted match — ACHV-03).

This is the final vertical slice and the highest-dependency plan: it consumes the 04-01 stat functions, the 04-03 `computeLifetimeStats`, and shares `match/+page.svelte` with 04-02 (hence Wave 3).

Purpose: Detect and celebrate personal records live on both views per D-03..D-08, ACHV-01..03.
Output: shared sync constants, `RecordOverlay`, record detection in `MatchStore` (+ test), `recordBadge` on the three win banners (D-08), and the overlay/badge wired into both routes.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| BroadcastChannel('neverman-record') → /display | Same-origin record events cross window boundary into the spectator view |
| record text → DOM (overlay + banner badge) | Record strings (may embed player names) rendered on both views |
| db.matches read at match start → preloadedRecords | Historical blobs feed the record baseline |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-11 | Spoofing | BroadcastChannel('neverman-record') message injection | mitigate | BroadcastChannel is same-origin only — no cross-origin injection possible (same control as Phase 2 T-02-01). The display handler treats only `{type:'record-event'}` shapes; unknown shapes are ignored. |
| T-04-12 | Tampering | DisplayStore corrupted by record event cast as MatchState (Pitfall 5) | mitigate | Use a SEPARATE channel `'neverman-record'` — the existing match-state handler never receives record events; the record handler never assigns to displayStore.state. |
| T-04-13 | Tampering (XSS) | record text / player names in RecordOverlay + record badge | mitigate | All strings via Svelte `{interpolation}` only — never `{@html}` (T-03-05). |
| T-04-14 | Denial of Service | record detection dispatching further actions → infinite loop | mitigate | `#detectRecords` only mutates `pendingRecords` and posts to the channel; it never calls `dispatch`/`reduce`. A unit test asserts no re-dispatch (RESEARCH security note). |
| T-04-15 | Tampering | preloadedRecords stale after mid-match profile deletion (Pitfall 4) | mitigate | loadRecords wrapped in try/catch; detection skips players absent from preloadedRecords or flagged guest — play continues uninterrupted. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No packages added this phase (RESEARCH Package Legitimacy Audit: not applicable). No install tasks exist. |

No auth/session/crypto boundaries in this static client-only PWA.
</threat_model>
