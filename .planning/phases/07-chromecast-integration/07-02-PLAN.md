<objective>
Define and test the trimmed `CastDisplayState` projection (D-05/D-07): a bounded snapshot containing exactly what the `/display` scoreboard components read, small enough to stay well under Cast's silent 64 KB message cap on long matches. TDD because this is a pure data transformation with a defined input (`MatchState` + pause state) and a defined output (`CastDisplayState`).

Purpose: A full `MatchState` (with full cross-match throw history in `players[].visits`) can exceed 64 KB on a long 4-player sets match and be silently dropped, failing exactly the long matches this app exists to score (D-06). The projection is the single source of the Cast payload shape, consumed by both the sender publish (Plan 05) and the receiver ingress (Plan 04).
Output: `src/lib/cast-types.ts` with `CastDisplayState`, `toDisplayState()`, `isValidCastState()`, and a `CastSnapshotMessage` wrapper type — all tested.
</objective>

<behavior>
The TDD test suite (`src/lib/cast-types.test.ts`, unit project) must drive these behaviors before implementation:

- **Field completeness:** Given a representative `MatchState` (2 players mid-leg), `toDisplayState()` returns an object whose shape, when fed to a `PlayerPanel`/`MatchHeader`, supplies every field those components read. The `/display` read-surface (confirmed from `PlayerPanel.svelte` + `+page.svelte` + `MatchHeader.svelte`):
  - top level: `config` (`{ startScore, outRule, legsToWin, setsEnabled, setsToWin }`), `players[]`, `activePlayerIndex`, `currentVisit`, `phase`, `legStartVisitIndex`
  - per player: `id`, `name`, `remaining`, `legsWon`, `setsWon`, `visits` (array of `Visit` = `{ darts, dartsAtDouble, bust, wasCheckout? }`)
  - pause (piggybacked, NOT part of MatchState): `pauseActive`, `pauseRemainingSeconds`
- **`isValidCastState` guard parity:** mirrors the existing `isValidMatchState` discipline — returns true only when `players` is a non-empty array AND `activePlayerIndex` is a number in `[0, players.length)`. Returns false for null/undefined/empty-players/out-of-range index. (Plan 04 calls this at the receiver ingress before assigning to display state.)
- **Pause passthrough (SYNC-03):** the `pauseActive`/`pauseRemainingSeconds` arguments appear verbatim on the output so the receiver runs its own local countdown from a single number.
- **Size bound (D-07):** construct a worst-case long match — 4 players, sets enabled, each player carrying a large `visits` history (simulate ~40 visits/player across legs) — and assert `new TextEncoder().encode(JSON.stringify(toDisplayState(...))).length` is under 32768 bytes. This test is the executable contract for D-07. If it fails, the projection must trim `visits` further.

**Projection design decision the executor must make and record in the SUMMARY (Claude's Discretion per CONTEXT.md):** how far to trim `players[].visits`. The display reads `visits` for three things: `recentVisitsWithScores` (only the last 4), `legAverage` (slice from `legStartVisitIndex`), and `matchAverage` (all visits — but the live display's `matchAverage()` already only reflects the current leg per its documented limitation). Therefore the projection MAY scope each player's `visits` to the current leg only (`visits.slice(legStartVisitIndex[id])`) and re-base `legStartVisitIndex` to 0, OR keep full `visits` if the size test passes. Pick the smallest shape that (a) keeps `recentVisitsWithScores` correct for the current leg, (b) keeps `legAverage`/`matchAverage` numerically identical to what `/display` shows today for the in-progress leg, and (c) passes the 32 KB test. Record the chosen trim and its rationale.
</behavior>
