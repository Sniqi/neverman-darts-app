<behavior>
TDD suite fills `src/lib/cast-sender.test.ts` (the RED scaffold from Plan 01), unit project. The Cast SDK is not present in Node, so the test assigns a mock onto `(globalThis as any).cast` and `(globalThis as any).chrome` BEFORE importing/exercising the manager (the same global-mock approach the existing unit tests use). Drive these behaviors:

- **Initial availability (CAST-04):** a freshly constructed manager has `castAvailable === false` and `activeSession === null`. Calling `init(appId)` and then invoking the captured `window.__onGCastApiAvailable(false)` (non-Chrome path) leaves `castAvailable === false`. Invoking it with `true` sets `castAvailable === true`.
- **App ID + auto-join (CAST-05, SETUP-02):** when `__onGCastApiAvailable(true)` fires, the manager calls `CastContext.getInstance().setOptions(...)` with `receiverApplicationId === appId` (the value passed to `init`, NOT a literal) and `autoJoinPolicy === chrome.cast.AutoJoinPolicy.ORIGIN_SCOPED`. Assert via the mock's recorded `setOptions` argument.
- **Load-order safety (Pitfall 2):** `init()` assigns `window.__onGCastApiAvailable` BEFORE appending the SDK `<script>`. Assert the callback is a function immediately after `init()` returns (the test inspects `window.__onGCastApiAvailable` without the real script ever loading).
- **Session transitions (CAST-02):** firing a mock `SESSION_STATE_CHANGED` event with `sessionState = SESSION_STARTED` sets `activeSession` to the mock's current session; `SESSION_ENDED` clears it to null.
- **Resume signal (CAST-06):** firing `SESSION_STATE_CHANGED` with `sessionState = SESSION_RESUMED` both sets `activeSession` AND raises the one-shot resume signal (carrying the device friendlyName for the toast body "Überträgt weiter auf: <Gerät>"); the signal reads true exactly once then resets (so the toast shows once per resume).
- **sendSnapshot guard:** with `activeSession === null`, `sendSnapshot(payload)` is a no-op (does not throw). With an active mock session, it calls `session.sendMessage(CAST_NS, payload)` exactly once; a throw inside `sendMessage` is swallowed (match play must continue — additive/non-fatal contract).
</behavior>
