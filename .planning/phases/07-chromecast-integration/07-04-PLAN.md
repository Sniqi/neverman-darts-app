<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| same-LAN Cast sender ‚Üí receiver UI state | Any sender on the Cast session can post a `CAST_NS` message; the receiver applies it to UI state |
| Chromecast ‚Üí gstatic CDN | Receiver SDK loaded over HTTPS from Google's CDN |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-IV | Tampering | malicious/malformed `CastDisplayState` from a same-LAN sender | mitigate | `receiveSnapshot` runs `isValidCastState(msg)` before applying to state (V5 Input Validation, ASVS L1). Home-use risk is low but the guard is cheap. |
| T-07-05 | Tampering | receiver SDK loaded on a normal browser (inertness) | mitigate | SDK is inert without `context.start()`; `start()` is gated behind `isCastReceiverContext()` (D-02) ‚Äî no observable side effect on tablet/PC. |
| T-07-04b | Information Disclosure | receiver page is public on GitHub Pages | accept | Receiver renders only scoreboard data already visible in the room; no PII, private unpublished Cast app, LAN-local. |
</threat_model>
