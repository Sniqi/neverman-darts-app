# Phase 7 — UI Design Contract

> Visual and interaction contract for the Chromecast Integration phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** This phase is purely additive. The spec covers only the Cast-specific UI additions: (1) the Cast status UI in `/match`, (2) the TV overscan margin treatment on the receiver, (3) the SESSION_RESUMED resume toast, and (4) the RECV-05 remaining-score transition. The existing `/display` scoreboard components (MatchHeader, PlayerPanel, PauseOverlay, RecordOverlay, LegWinBanner, MatchWinDisplay, IdleScreen) are reused unchanged.

## Component Contracts

### 1. Cast Status UI in `/match` (CAST-01, CAST-02, CAST-03, CAST-04, D-12, D-13)

**Placement:** Inside `SpectatorChooser.svelte`'s chooser menu (the bottom-sheet / landscape popover that opens when the monitor icon button is tapped). Add a Cast row as a new item in the chooser list, below the existing two options. Do NOT add a second fixed button or modify the `.chooser-icon-btn` position.

Rationale: The chooser menu is the established "display mode" surface in `/match`. A Cast option belongs there alongside "Zweites Fenster" and "Anzeige im Vollbild". This keeps the dartboard and scoring area completely uncluttered (D-12: "away from the dartboard").

**Cast row — when Cast is available (verfügbar or verbindet or verbunden):**

```
[ cast-icon 20×20 ]  Auf Chromecast übertragen
```

- Width: full-width, `min-height: 48px` — identical to existing `.chooser-action-btn`
- Background: `#111318`, border: `1px solid #333`, border-radius: `6px`
- Font: 16px / weight 400 / color `--text`
- Icon: inline SVG cast icon (Chromecast screen-with-wifi-waves shape), `width="20" height="20"`, `stroke="currentColor"`, `opacity: 0.7` in default state
- Active state (`:active`): `background: #22242d` (matches `.chooser-action-btn:active`)

**"Überträgt auf: \<Gerät\>" indicator — shown only when verbunden (connected):**

Displayed directly below the "Auf Chromecast übertragen" row within the chooser menu, as a compact status line (not a separate button):

```
● Überträgt auf: Wohnzimmer TV
```

- Font: 14px / weight 400 / line-height 1.4
- Color: `--accent` (`#e8a020`) for the dot + device name; `rgba(240,240,240,0.6)` for "Überträgt auf:"
- Dot: unicode `●` U+25CF, 10px, accent color
- Padding: `4px 16px` — matches chooser action button horizontal padding
- No border, no background — inline status line only

**`<google-cast-launcher>` web component placement:**

The `<google-cast-launcher>` element is the tap target that opens the native Cast dialog (for device selection and stopping cast). Embed it invisibly within the "Auf Chromecast übertragen" row by setting `display: none` on the web component and intercepting the `pointerdown` on the row button to programmatically invoke `CastContext.getInstance().requestSession()`. This avoids the web component's default visual chrome conflicting with the app's design. Alternatively if the web component's button must be rendered: size it to `20×20px` and apply `--cast-button-icon-color: currentColor` so it inherits the row's text color.

**Visibility rules (D-13):**

- If `cast(CastContext)` is unavailable (non-Chrome browser, or `window.chrome?.cast` not present after SDK load attempt): the Cast row is completely absent from the chooser menu — `{#if castAvailable}` guard. No disabled state, no placeholder.
- If Cast is available but no devices found: show the row normally (verfügbar state). The SDK handles "no devices" messaging in its own dialog.

**State label (below cast row, when available):**

| `castState` | Label shown |
|-------------|-------------|
| `NOT_CONNECTED` | _(no label — row alone suffices)_ |
| `CONNECTING` | `Verbindet…` — 14px, `rgba(240,240,240,0.6)` |
| `CONNECTED` | replaced by "Überträgt auf:" line (above) |
| `NO_DEVICES_AVAILABLE` | _(row hidden per D-13 policy if truly unavailable, or shown as verfügbar if devices could appear)_ |

**Z-index:** The SpectatorChooser menu is at z-index 50. Cast additions are inside it — no new z-index layer needed.

### 2. Receiver Overscan Contract (D-11, RECV-01)

**Trigger:** `isCastReceiverContext()` returns true → add class `cast-receiver` to the `/display` route root element (`<div class="display-root">` in `+page.svelte`).

**Margin values:**

| Property | Value | Rationale |
|----------|-------|-----------|
| `padding` on `.cast-receiver` | `96px` all edges | 10% of 960px (half of 1920px) = 96px, rounded to nearest 8. Keeps content within the overscan-safe zone on TV sets at 1920×1080. |

Implementation: add a modifier class. Do not change the base `.display-root` styles.

```css
.display-root.cast-receiver {
  padding: 96px;
}
```

The `panels-grid` and `MatchHeader` already use `height: 100dvh` and `flex: 1 1 0` — the padding on the root will contract the available space and the inner components will scale correctly via their existing `clamp()` and `flex` sizing. No changes needed inside the child components.

**PC second window and tablet fullscreen:** unaffected — `cast-receiver` class is never applied in those contexts (SYNC-04 protected).

### 3. "Verbindung wiederhergestellt" Resume Toast (CAST-06, D-14)

**Trigger:** Cast SDK fires `SESSION_RESUMED` event on the sender after a page reload rejoins an active session.

**Appearance:**

```
┌─────────────────────────────────────────┐
│▌ Verbindung wiederhergestellt           │
│  Überträgt weiter auf: Wohnzimmer TV    │
└─────────────────────────────────────────┘
```

| Property | Value |
|----------|-------|
| Position | Fixed, bottom-right: `bottom: 80px; right: 16px` (above the SpectatorChooser button at `bottom: 16px`) |
| Width | `min-width: 240px; max-width: 320px` |
| Background | `#1e2027` (`--surface`) |
| Border | `1px solid var(--line-strong)` |
| Left accent stripe | `4px solid var(--accent)` (`#e8a020`), `border-radius: 0` on left side |
| Border radius | `var(--radius-md, 12px)` — follows existing panel radius |
| Padding | `12px 16px` (top/bottom 12px = between `--space-sm` and `--space-md`; left 16px = `--space-md`) |
| Heading line | "Verbindung wiederhergestellt" — 16px / weight 600 / `--text` |
| Body line | "Überträgt weiter auf: \<Gerät\>" — 14px / weight 400 / `rgba(240,240,240,0.6)` |
| Shadow | `var(--shadow-raise, 0 2px 10px rgba(0,0,0,0.4))` |
| Z-index | 35 (above SpectatorChooser trigger at z-40 is wrong — place it at z-index 45, below the chooser menu at z-50 but above the fixed trigger at z-40) |

**Animation:**

- Enter: `opacity 0→1, translateY(8px→0)` over `200ms ease-out`
- Exit: `opacity 1→0` over `150ms ease-in` (no translate on exit)

**Duration:** 3500ms visible, then auto-dismiss (fade out). No user interaction required — purely informational. No dismiss button.

**Svelte implementation note:** Use a component with `{#if showResumeToast}` gated on the state. Use `in:fade` / `out:fade` from `svelte/transition` with the above durations or implement the CSS animation directly. Keep the component in `/match` only (sender context).

### 4. RECV-05 Remaining-Score Transition (RECV-05, D-14)

**Target element:** `.remaining-score` in `src/ui/display/PlayerPanel.svelte`.

**Current state:** No CSS transition on the `remaining-score` element.

**Contract:** Add a CSS transition so the score value change is visible as a smooth update rather than an instant jump, drawing the eye of the spectator across the room.

| Property | Value |
|----------|-------|
| Transition property | `color` |
| Duration | `300ms` |
| Easing | `ease-out` |
| Flash color | On update: momentarily `#ffffff` (full white), returning to the existing color (`#f0f0f0` inactive / `#ffffff` active) |

**Implementation approach:** Use a Svelte `$effect` that detects when `liveRemaining` changes, temporarily applies an `.updating` class that overrides the color to `#ffffff`, then removes it after 300ms. The CSS transition handles the fade back.

```css
/* Add to PlayerPanel.svelte <style> */
.remaining-score {
  transition: color 300ms ease-out;
}

.remaining-score.updating {
  color: #ffffff;
}
```

This is non-breaking for the PC second window and tablet fullscreen paths — the transition is a purely visual enhancement that applies whenever `liveRemaining` changes, regardless of transport.

**Do not add** a `transform` (scale/translate) transition — it would cause layout shift inside the flex `name-score-row` and could mis-align the checkout suggestion. Color-only flash is sufficient and safe.

### 5. Receiver Loading Screen (RECV-02)

**Trigger:** Chromecast page loaded + `CastReceiverBridge.init()` called + no `receiveSnapshot()` call yet received.

**Reuse:** The existing `IdleScreen.svelte` component shows "Warte auf Match…" in the pre-match state. The loading state before first Cast message can reuse `IdleScreen` with no modification — it already handles `matchState === null`. No new loading component needed.

If a distinct Cast loading state is desired (to distinguish "no match started" from "waiting for Cast state to arrive"), add a prop or separate state branch:

| State | Display |
|-------|---------|
| `castReceiving` (between init and first snapshot) | `IdleScreen` — text: "Warte auf Spielstand…" |
| `idle` (no active match in IndexedDB either) | `IdleScreen` — existing text: "Warte auf Match…" |

Both use the same idle screen layout. The text difference is the only distinction (optional — the planner decides).

### 6. Receiver Idle / Disconnect Screen (RECV-03)

**Trigger:** Cast session ends or sender disconnects while the receiver is running.

**Reuse:** `IdleScreen.svelte` unchanged. Text shown: "Kein aktives Spiel" (from REQUIREMENTS.md RECV-03). Update `IdleScreen`'s status text prop or render conditionally based on disconnect reason.

| State | Heading | Body |
|-------|---------|------|
| Disconnected after active session | "Kein aktives Spiel" | _(none — heading alone suffices on TV)_ |
| Never connected / initial load | "Warte auf Match…" | _(none)_ |

Font scale for TV: `IdleScreen` uses 20px heading / 16px body — this is adequate at 1080p. No change needed.

## Interaction Contracts

### Touch targets

All new interactive elements must meet the 44×44px minimum touch target:
- Cast row in chooser menu: `min-height: 48px; width: 100%` — satisfies 44px minimum (matches existing `.chooser-action-btn`)
- Resume toast: no interactive elements, no touch target required

### Animations (all new in Phase 7)

| Element | Property | Duration | Easing | Direction |
|---------|----------|----------|--------|-----------|
| Resume toast enter | `opacity + translateY` | 200ms | ease-out | in |
| Resume toast exit | `opacity` | 150ms | ease-in | out |
| RECV-05 remaining-score flash | `color` | 300ms | ease-out | in→out |
| Cast row in chooser | `background` on `:active` | — | instant | n/a |

No other new animations. Do not add animation to the "Überträgt auf:" status line — it appears and disappears with the menu open/close, which the existing chooser animation handles.

### Keyboard / accessibility

- Cast row in chooser: standard `<button>` element, inherits focus ring from existing button styles
- Resume toast: `role="status"` or `aria-live="polite"` — auto-announced by screen readers without focus
- "Überträgt auf:" indicator: no special aria needed (visible text is sufficient; it's inside an open dialog)
- TV receiver: no keyboard or pointer interaction expected (TV is output-only in this use case)
